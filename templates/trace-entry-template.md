# 需求演进链模板

> 用于 `[[projects/trace]]` 的单个需求主题主链。默认优先续写已有主题，再考虑新开主链。

## TRACE-XXX 一句话概括这个需求主题

- **记录人**：优先写 `git user.name`；如果没有配置就删掉这一行。
- **原始意图**：
  - 先写真实业务目标，不要直接写最终实现口径。
- **收敛后的可执行需求**：
  - 写当前已经能够指导设计或开发的需求表达。
- **关键决策变化**：
  - **主收敛 / 主替换 / 主纠偏**：先写替换旧口径、范围收敛或实现纠偏。
- **最终范围**：
  - 先写当前真正生效的范围。
- **假设与未决项**：
  - 没有就写“暂无关键未决项”，或直接精简成 1 到 2 条。
- **关联页面**：
  - [[projects/requirements]]
  - [[projects/design/README]]
  - [[projects/decisions]]
- **迭代**：

### YYYY-MM-DD 或 YYYY-MM-DD HH:mm

- **本轮变化**：
  - **主收敛 / 主替换 / 主纠偏**：第 1 条先写本轮最关键的收敛、替换或纠偏。
- **当前实现口径**：
  - 写这轮结束后真正生效的口径。

## 使用说明

- 详细规则见 [[trace-writing-rules]]。
- `[[projects/trace]]` 不按日期分组，按需求主题维护主链。
- 同一主题默认只保留一条主链，后续变化优先写新的迭代块。
- 需要标注记录人时，优先使用当前仓库的 `git user.name`。
- 默认日期写到天；只有同一天多次收敛需要区分时，才升级到分钟。
- 原始意图、关键决策变化、最终范围不要互相混写。
- 修补性需求要明确标出来，但不要把它抬升成原始业务目标。
- 关联页面优先列需求、设计、决策、开发这几类真正会回看的主页面。
- 推荐把字段名和短标签加粗，但不要把整段正文整段加粗。
