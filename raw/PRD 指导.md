# PRD 指导

这是一份面向新手的 PRD 写作源稿。沉淀版见 [[concepts/prd-writing]]，摘要卡见 [[articles/2026-04-13-prd-writing-guide]]。

## 1. PRD 到底是什么

PRD，Product Requirements Document，产品需求文档。

它不是灵感记录，也不是“老板一句话的展开版”，而是把一个业务问题翻成团队可执行、可验收的需求文档。

一个合格的 PRD，至少要回答清楚这些问题：

- 为什么要做
- 给谁做
- 要解决什么问题
- 具体做什么
- 本期不做什么
- 做成什么样算完成

可以把它理解成：

**PRD = 业务目标 + 用户场景 + 功能方案 + 业务规则 + 验收标准**

## 2. 新手最容易犯的 5 个问题

### 2.1 只写“要做什么”，不写“为什么做”

比如只写：

- 增加会员签到功能

但没写：

- 为什么要加
- 目标是什么
- 解决什么问题

这样研发和设计会不知道重点，最后做出来可能偏掉。

### 2.2 写成功能清单，不写用户流程

比如罗列：

- 有按钮
- 有弹窗
- 有列表
- 有导出

但没有说明用户是怎么一步步操作的。

### 2.3 规则不完整

例如：

- 用户可以提交申请

但没写：

- 谁可以提交
- 提交失败怎么办
- 重复提交怎么办
- 审核中能不能修改
- 是否有权限限制

### 2.4 没有边界

不写“本期不做什么”，项目就容易无限膨胀。

### 2.5 没有验收标准

最后上线前，大家会争论：

- 这个算不算做完
- 这个是不是 bug
- 这个是不是需求变更

## 3. 最稳妥的 PRD 写法

最适合新手的，不是超长文档，而是“六段式 PRD”。

### 3.1 文档信息

这一部分回答：这份文档是谁写的、是什么版本、对应什么需求。

建议写清：

- 需求名称
- 版本号
- 作者
- 创建日期
- 相关人员

### 3.2 背景与目标

这一部分回答：为什么做。

建议写清：

- 业务背景
- 当前问题
- 本次目标
- 预期收益

示例：

> 当前用户在下单后无法实时查看配送进度，导致客服咨询量较高。  
> 为提升用户体验、减少客服压力，计划新增“订单物流跟踪”功能。  
> 本次目标是让用户在订单详情页直接查看当前配送状态和历史节点信息。

这一段不要写空话，少写“提升体验”“赋能业务”这种泛词，多写具体问题。

### 3.3 用户与场景

这一部分回答：谁在什么情况下会用。

建议写：

- 目标用户是谁
- 使用场景是什么
- 用户痛点是什么

示例：

> 目标用户：已下单用户  
> 使用场景：用户支付完成后，想知道商品当前配送到哪一步  
> 用户痛点：无法及时获取物流状态，只能等待短信或咨询客服

你可以用一句很实用的话来检验：

**“这个功能是哪个用户，在什么场景下，遇到了什么问题，所以需要什么能力？”**

### 3.4 需求范围

这一部分回答：本次做什么，不做什么。

建议拆成两块：

#### 本次需求范围

- 订单详情页展示物流状态
- 展示物流时间轴
- 异常状态提示
- 支持刷新物流信息

#### 本期不做

- 不支持短信提醒
- 不支持物流地图实时轨迹
- 不支持多平台物流比价

这一段非常关键。新手写 PRD 最容易漏掉“不做什么”，但这恰恰最能体现专业度。

### 3.5 功能详细说明

这一部分回答：具体怎么做。

这是 PRD 的核心。建议按“功能点”逐条写，每个功能点都尽量写成固定格式：

- 功能名称
- 功能说明
- 触发条件
- 处理逻辑
- 异常情况
- 页面/字段说明
- 权限限制

例如：

#### 功能点 1：订单详情页展示物流入口

**功能说明**  
用户在已发货订单的详情页，可查看“查看物流”入口。

**触发条件**  
订单状态为“已发货”“配送中”“已签收”时展示。

**处理逻辑**

- 未发货：不展示入口
- 已发货：展示入口，点击进入物流详情页
- 若物流信息获取失败：展示“暂时无法获取物流信息，请稍后重试”

**异常情况**

- 接口超时：提示加载失败
- 无物流单号：展示“暂无物流信息”

**权限限制**

- 仅订单所属用户可查看

像这样写，研发、测试、设计都会很舒服。

### 3.6 验收标准

这一部分回答：做成什么样算完成。

建议按“场景 + 预期结果”写。

例如：

- 当订单状态为“已发货”时，订单详情页展示“查看物流”入口
- 点击入口后，进入物流详情页，成功展示最新物流状态
- 当物流接口返回失败时，页面展示错误提示
- 当用户查看非本人订单时，无权限查看物流信息
- 页面在移动端正常展示，不出现布局错乱

验收标准越具体，后面扯皮越少。

## 4. 一个专业 PRD 的标准结构

你可以直接照这个结构写：

### 1. 文档信息

- 需求名称
- 版本号
- 作者
- 创建日期
- 相关人员

### 2. 背景与目标

- 背景
- 当前问题
- 目标
- 指标

### 3. 用户与场景

- 目标用户
- 使用场景
- 用户痛点

### 4. 需求范围

- 本次要做
- 本次不做

### 5. 功能方案

- 功能点 1
- 功能点 2
- 功能点 3

### 6. 业务规则

- 状态流转
- 权限控制
- 异常处理
- 字段规则

### 7. 页面与交互

- 页面入口
- 页面元素
- 交互说明
- 提示文案

### 8. 数据与埋点

- 关键字段
- 关键操作日志
- 埋点需求

### 9. 验收标准

- 功能验收
- 异常验收
- 权限验收

### 10. 附录

- 流程图
- 原型图
- 参考链接

## 5. 写 PRD 时最实用的思考顺序

别一上来就写正文。先按下面顺序想，会顺很多。

### 第一步：先说清楚问题

先写一句话：

> 现在谁在什么场景下遇到了什么问题？

比如：

> 用户下单后不知道商品配送到哪一步，需要频繁咨询客服。

### 第二步：再定义目标

> 我们想通过什么方式解决这个问题？

比如：

> 通过在订单详情页提供物流跟踪能力，让用户自助查询配送进度。

### 第三步：再拆成功能

> 为了实现这个目标，需要哪些功能？

比如：

- 物流入口
- 物流详情页
- 异常提示
- 状态刷新

### 第四步：补规则

> 这些功能分别在什么条件下展示？失败怎么办？谁能看？

### 第五步：补验收

> 什么情况下算成功？什么情况下算异常？

这套顺序比直接套模板更重要。

## 6. 专业感最强的 4 个细节

### 6.1 定义清楚

像这些词，最好都定义一下：

- 新用户
- 活跃用户
- 已支付
- 已完成
- 会员
- 有权限

因为不同团队理解可能不一样。

### 6.2 状态写清楚

如果需求涉及流程，一定要写状态流转。

例如：

- 待提交
- 审核中
- 审核通过
- 审核拒绝
- 已关闭

并说明：

- 怎么进入这个状态
- 谁能操作
- 能不能回退

### 6.3 异常写清楚

专业 PRD 很大的区别就在这里。

很多人只写正常流程，但真实项目里，异常情况特别多。

你至少要考虑：

- 接口失败
- 数据为空
- 重复操作
- 权限不足
- 网络异常
- 并发冲突
- 状态不一致

### 6.4 文案写清楚

少写：

- 适当提示
- 友好提示
- 合理校验

多写：

- Toast 提示“提交成功”
- 弹窗提示“确认删除后不可恢复”
- 输入超过 50 字时提示“最多输入 50 字”

- xxx

## 8. 如果你想让 PRD 真正指导设计和开发

PRD 不是写给自己看的，是给团队协作用的。

在这套文档库里，建议按下面这条链路走：

- 先把原始想法收成 PRD
- 再把 PRD 翻成 `projects/requirements.md`
- 设计页只接收已经收敛过的需求
- 如果出现多个方案或规则争议，先写进 `projects/decisions.md`
- 如果需求在推进过程中发生变化，再写进 `projects/trace.md`
- 最后把稳定结论回收到知识库层

这样，PRD 才不会只停留在“看起来很完整”的文档，而是真的能推动设计和开发。

## 9. 一个简单判断标准

写完后，你可以自己检查这 7 个问题：

- 为什么做，是否说清楚了？
- 目标用户是否明确？
- 本次范围是否明确？
- 主流程是否完整？
- 业务规则是否完整？
- 异常情况是否考虑到了？
- 验收标准是否可测试？

如果这 7 个问题都能答上来，这份 PRD 基本就合格了。

## 10. 对小白最重要的一句话

**专业 PRD 不是“写很多”，而是“写清楚”。**

你刚开始完全没必要追求几十页的大文档。先把一个小需求写清楚，比堆很多术语更重要。

你可以先用这个思路：

**背景 → 用户场景 → 目标 → 范围 → 功能 → 规则 → 验收**

把这条链路写顺了，PRD 自然会越来越专业。

如果你愿意，下一步可以直接拿一个真实小需求，按这套方法拆成可执行 PRD。
