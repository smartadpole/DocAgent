# `[[projects/trace]]` 记录规则

这页是 `[[projects/trace]]` 写法的唯一主入口。

`[[projects/trace]]` 本身只负责承接当前生效的需求演进链；怎么分主题、怎么续写、什么时候新开主链、什么时候只补迭代，统一以这页为准。

## 这页负责什么

- 定义 `[[projects/trace]]` 的记录单位和默认骨架。
- 定义 trace 主链与迭代块的关系。
- 定义 trace 和 `[[log]]`、[[projects/decisions]]、[[projects/memory/README]] 的边界。
- 给出续写旧主题还是新开主题的最小判断顺序。

## 这页不负责什么

- 不承接逐轮历史正文，那是 `[[projects/trace]]` 的职责。
- 不重复维护模板正文，那是 [[templates/trace-entry-template]] 的职责。
- 不代替项目总流程，那是 [[WORKFLOW]] 的职责。

## 核心标准

- `[[projects/trace]]` 记录的是“需求如何收敛成当前实现口径”，不是对话逐句转录。
- 记录单位是“需求主题主链”，不是“每天一条”，也不是“每次对话一条”。
- 同一主题默认只保留一条主链，后续变化优先写成新的迭代块，而不是平行复制第二条主链。
- 主链优先回答：原始意图是什么、后来怎么变化、现在真正生效的范围是什么。
- 迭代块优先回答：这一轮新增了什么约束、修补了什么偏差、最终口径变成了什么。
- trace 重点看需求主题和当前有效口径，不按日期分组。

## 和相邻页面的边界

- `[[log]]` 记录这轮对话在推进什么主题、做了哪些关键动作。
- [[projects/trace]] 记录需求怎样从原始意图、约束变化、修补性需求一路收敛到当前实现口径。
- [[projects/decisions]] 只记录真正需要拍板的冲突和取舍，不重复抄整条演进链。
- [[projects/memory/README]] 只记录项目长期稳定背景，不承接某一轮需求如何变化。

## 默认骨架

每个 trace 主题默认包含：

- 主题
- 原始意图
- 收敛后的可执行需求
- 关键决策变化
- 最终范围
- 假设与未决项
- 关联页面
- 迭代

每个迭代块默认包含：

- 日期
- 记录人
- 角色（可选）
- 本轮变化
- 当前实现口径
- 必要时补充风险、假设或后续动作

## 记录人与日期怎么写

- `记录人` 默认写在迭代块里，不再默认挂在整个主题主链顶层。
- 只有当某条主链明确需要说明“当前由谁维护”时，才额外写主题级责任信息；默认不写单个主题级 `记录人`。
- 迭代块里的 `记录人` 优先写当前仓库的 `git user.name`；如果没有配置就留空或删掉这一行，不回退到本机用户名。
- 如果需要表达参与视角，可以在迭代块里追加 `角色`，例如“产品”“研发”“agent”“用户确认”；没有必要时不强行补。
- trace 主链本身不按日期分组，但每个迭代块都必须带日期。
- 默认日期格式统一写 `YYYY-MM-DD`；如果同一天内有多次需要区分的收敛动作，可以升级成 `YYYY-MM-DD HH:mm`。
- 只有当分钟级时间真的有助于区分多轮收敛时，才写到分钟；不要为了形式把每个迭代都写得过细。
- 如果主链本身发生了明显重构、改名或口径替换，除了 frontmatter 的 `updated`，也要在最新迭代里留下明确日期锚点。

## 什么时候续写旧主题

优先续写已有主题，而不是新开主链，如果这轮变化满足下面大部分条件：

1. 原始业务目标没有变化，还是在推进同一个主题。
2. 新增内容主要是澄清约束、范围收敛、实现纠偏或补充落地口径。
3. 读者以后回看时，更希望在同一条主链里看到“它是怎么一步步收敛的”。
4. 如果强行拆成新主题，会让同一需求的前因后果散到两处。

## 什么时候新开主题

只有满足下面任一条件，才新开一条 trace 主链：

- 原始业务目标已经变化，不能再用原主题那句话概括。
- 当前讨论已经切换成另一组需求，而不再是原主题的收敛或纠偏。
- 继续写在旧主题里，会让“当前生效口径”变得模糊。
- 回看时如果不拆，会明显增加定位成本。

## 修补性需求怎么写

- 如果本轮变化主要是在修复 agent 或实现过程引入的偏差，要明确标成修补性需求或实现纠偏。
- 修补性需求要写进 `关键决策变化` 或迭代块，但不要把它抬升成 `原始意图`。
- 如果修补最终改变了当前有效行为，`最终范围` 和 `当前实现口径` 也要同步更新。

## 怎么把重点写出来

- 主题标题先写需求对象和收敛方向，不写成“补充 trace”“继续调整”这类空标题。
- `最终范围` 先写当前真正生效的口径，再写补充说明。
- `关键决策变化` 里先写替换旧口径或真正改变实现方向的变化，后写支撑细节。
- 迭代块里的第 1 条变化默认写本轮最关键的收敛、替换或纠偏。

## 怎么做文字渲染

- 合适：把字段名统一写成加粗，例如 `- **原始意图**：`、`- **关键决策变化**：`、`- **最终范围**：`、`- **关联页面**：`、`- **记录人**：`、`- **角色**：`。
- 合适：把关键变化里的短标签写成加粗，例如 `**主收敛**：`、`**主替换**：`、`**主纠偏**：`。
- 合适：如果某个迭代块里第 1 条变化明显最关键，只加粗其中 2 到 6 个字的核心短语，不把整条加粗。
- 不合适：把整段 `原始意图`、`最终范围` 或整条迭代说明整段加粗，这会让真正重点变得不明显。
- 不合适：同一条主链里同时大量混用加粗、代码样式和引号，导致视觉层级混乱。
- 默认原则和 `[[log]]` 保持一致：字段名可加粗，正文不整段加粗，短标签可加粗，重点优先靠顺序而不是花哨样式突出。

## 写 trace 时的最小顺序

1. 先判断这轮是不是已经进入项目推进，而不只是资料收集或闲聊。
2. 再判断它属于哪个已有需求主题，还是已经形成新主题。
3. 先回看原始意图，再写这轮变化，不要直接从最后一句用户话反推。
4. 明确区分原始目标、后续新增范围、修补性需求和最终生效口径。
5. 如果这轮需要记录参与人、确认人或 agent 贡献，把这些信息写到对应迭代块，不要抬到整条主链顶层。
6. 最后再决定是续写迭代块，还是新开主链。

默认模板见 [[templates/trace-entry-template]]。
