# Obsidian 文档系统设计

- 来源：多轮对话沉淀
- 日期：2026-04-09
- 类型：设计研究

## 一句话总结

这个系统的目标，是用 Obsidian 作为前台，用 Git 作为审计，用文档层级和规则编排，把软件开发文档管理从“半自动的人读人写”逐步升级为“可自动运行的文档操作系统”。

## 核心判断

### 1. 总方向正确，不需要推翻重来

当前骨架已经具备这些层次：

- 入口层：[[README]] / [[INDEX]]
- 治理层：[[governance/README]] / [[AGENTS]] / [[WORKFLOW]]
- 共享背景层：[[BRAIN]]
- 规则层：[[POLICY]]
- 项目层：`projects/`
- 项目记忆层：`projects/memory/`
- 知识沉淀层：`articles/` / `concepts/` / `indexes/`
- 历史层：`archive/` / [[log]]

这说明问题不是“有没有结构”，而是“层间边界是否足够硬、是否足够机器可读”。

### 2. 目前适合的运行模式是单库、强规范、弱权限、强链接、可编排

当前场景的关键约束是：

- 几个人协作
- 没有账号体系
- 以 Obsidian 为主界面
- 先半自动，后自动化

因此最合适的不是做复杂平台，而是继续把单一 vault 做成一个强约束文档系统。

### 3. 不需要先做账号系统

这类场景下，Git 已经足够承担大部分：

- 身份痕迹
- 变更审计
- 回滚恢复
- 协作同步

所以系统设计不应该围绕“人账号”展开，而应该围绕：

- `local`
- `shared`
- `project`
- `policy`

这些作用域展开。

### 4. 最大问题不是“目录乱”，而是层间边界还不够硬

主要集中在三点：

- [[BRAIN]] 需要只保留共享背景
- [[POLICY]] 需要承接规则和优先级
- `projects/memory/` 需要承接项目级稳定记忆
这些职责现在分别由 [[BRAIN]]、[[POLICY]] 和 `projects/memory/` 承接，且治理页已经物理收口到 `governance/`。

### 5. memory 先做文档化，不急着上数据库

未来要加入分层 memory，可以先把它做成文档化 memory，而不是一开始就上独立存储引擎。

也就是说：

- 先让文档成为 memory 的承载层
- 再让 agent 读取这些文档
- 最后才考虑接入外部 memory 工具

## 现有落点

这套设计现在已经落到一组明确页面里：

- [[governance/README]]：治理层入口
- [[BRAIN]]：共享背景
- [[POLICY]]：规则、优先级和路由边界
- `projects/memory/`：项目级稳定记忆
- [[projects/decisions]]：项目阶段的关键取舍
- [[projects/README]]：项目运行主入口

## 结构建议

### 1. 治理层和内容层分开

建议把根层语义继续收口成：

- [[README]] / [[INDEX]]：入口
- [[governance/README]] / [[AGENTS]] / [[WORKFLOW]] / [[BRAIN]] / [[POLICY]]：治理和背景
- `projects/`：运行层
- `articles/` / `concepts/` / `indexes/`：沉淀层
- `archive/` / [[log]]：历史层

### 2. BRAIN 只保留共享背景

[[BRAIN]] 适合回答：

- 已确认的共同前提是什么
- 后续默认要带入哪些背景
- 哪些项目共识不需要重复解释

不适合继续长期混入：

- 自动沉淀规则
- 冲突升级制度
- 优先级覆盖顺序

这些更适合进入规则层或决策层。

### 3. 决策和制度单独收口

思维碰撞发生时，不能直接覆盖共享背景。

更稳的做法是：

- 先进入项目决策
- 决策明确后，再回写共享背景或规则层

### 4. 对小团队的建议

#### 1. 继续保持“单库”

不要拆成多个仓库，不要上来做多账号权限系统。

#### 2. 继续保持“强规范”

规则和入口要少而稳定，避免每次任务都重新解释一遍系统怎么用。

#### 3. 继续保持“半自动到自动化”的演进路径

先让人决定做哪一项，再逐步把可判断的部分交给 agent。

## 未来分层 memory 的落点

如果未来正式推进分层 memory，建议遵循这个顺序：

1. 先在文档里定义层次和路由
2. 再定义共享背景和 policy 的边界
3. 再定义项目 memory 的承载位
4. 最后再考虑接 Mem0 / Zep / 向量检索等工具

## 当前判断

如果目标是先把系统做稳：

- 维持单库
- 保持小团队协作
- 让 [[BRAIN]]、[[POLICY]] 和 `projects/memory/` 分层
- 用 `projects/decisions.md` 收口冲突

如果目标是继续往自动化走：

- 先把文档层做成可读的状态机和路由系统
- 再把可重复的判断逐步交给 agent
