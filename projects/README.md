# Projects

这里放活跃软件研发项目的运行文档。

## 和知识库模式的关系

- 知识库模式是底座，负责长期沉淀和复用。
- 研发模式是叠加层，负责项目推进和阶段管理。
- 项目里的稳定结论，最终要回写到知识库层。
- 项目里的上下文、一次性决策和阶段记录，留在项目层即可。

## 研发推进的关键原则

- 项目主页是单一项目的主入口，状态、范围和当前判断优先看这里。
- 代码仓库是实现真相源，vault 是决策真相源和上下文真相源。
- 先定义问题、目标、范围和成功标准，再进入实现，避免边做边漂。
- 需求、设计、决策、排障、发布和复盘都要留下可回看的文字痕迹。
- 需要复用的结论，最后都要回写到 `articles/`、`concepts/` 或 `indexes/`。
- 一次性上下文留在项目页，长期稳定知识抽回知识库层。
- 流程控制保持人工判断，不依赖隐藏自动化，这样更适合小项目推进。
- 如果文档之间有冲突，以项目主页为准，再向下修正其他页面。

## 这层负责什么

- 需求收集
- 方案设计
- 决策记录
- 实施跟踪
- 发布说明
- 事故和复盘

## 极简版本

如果你只有一个小项目，最小只需要这两个东西：

- `projects/<项目名>/README.md`
- 根目录 `log.md`

其余文件都按需添加，不要为了完整性先建空文件。

## 这层不负责什么

- 不把项目上下文直接混进 `articles/`、`concepts/` 或 `indexes/`
- 不把可复用知识长期留在项目页里
- 不做自动流控，状态由人读项目主页后手动判断

## 推荐结构

每个项目建议单独一个子目录，例如：

```text
projects/
  my-service/
    README.md
    requirements.md
    design.md
    decisions.md
    worklog.md
    releases.md
    incidents.md
```

如果项目数量变多，再补一个 `projects/INDEX.md` 汇总活跃项目入口。单项目场景下，这一步可以先不做。

## 典型流程

1. 立项时先建项目目录和 `README.md`。
2. 可以用 `templates/project-template.md` 生成项目主页，也可以直接手写 `README.md`。
3. 收集需求、会议纪要、Issue、PR 和外部资料。
4. 先写清目标、范围、非目标和成功标准。
5. 再补设计、决策和实施记录。
6. 发布后写发布说明和复盘。
7. 稳定下来的结论回写到知识库层。

## 手动流控

- 建议状态词：`idea`、`active`、`blocked`、`review`、`released`、`archived`
- 每次切状态前先读项目主页
- 如果多个文档冲突，以项目主页为准
- 如果项目结束，把可复用内容提炼回知识库，把项目特有内容保留在项目目录或归档层
