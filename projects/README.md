# Projects

这里放活跃软件研发项目的运行文档。

## 和知识库模式的关系

- 知识库模式是底座，负责长期沉淀和复用。
- 研发模式是叠加层，负责项目推进和阶段管理。
- 项目里的稳定结论，最终要回写到知识库层。
- 项目里的上下文、一次性决策和阶段记录，留在项目层即可。

## 极简版本

如果你只有一个小项目，最小只需要这两个东西：

- `projects/README.md`
- 根目录 `log.md`

其余文件都按需添加，不要为了完整性先建空文件。

## 这层不负责什么

- 不把项目上下文直接混进 `articles/`、`concepts/` 或 `indexes/`
- 不把可复用知识长期留在项目页里
- 不做自动流控，状态由人读项目主页后手动判断

## 推荐结构

当前设计是一个 `wiki` 只服务一个项目，所以 `projects/` 本身就是这个项目的运行层，例如：

```text
projects/
  README.md
  requirements.md
  design.md
  decisions.md
  worklog.md
  releases.md
  incidents.md
```

这些文件都按需创建，不需要一次性铺满。

## 常用文件

- `projects/README.md`：项目主页，也是单一项目的总入口。这里放目标、状态、范围、关键链接、当前判断和下一步。
- `requirements.md`：需求说明。适合放用户问题、业务目标、范围、非目标、验收标准。
- `design.md`：方案设计。适合放架构、模块划分、接口、数据流、技术取舍。
- `decisions.md`：决策记录。适合放为什么选这个方案、不选另外一个方案、有哪些约束。
- `worklog.md`：实施记录。适合放每天推进了什么、遇到什么问题、怎么解决、还剩什么阻塞。
- `releases.md`：发布说明。适合放上线范围、变更点、验证方式、回滚方案。
- `incidents.md`：事故和复盘。适合放故障现象、影响范围、根因、修复动作和后续改进。

## 研发阶段怎么落到文件

- `调研/立项`：先写 `projects/README.md`，必要时再补 `requirements.md`
- `设计/评审`：补 `design.md`，关键取舍写进 `decisions.md`
- `开发/联调`：持续更新 `projects/README.md` 的状态和下一步，复杂过程再补 `worklog.md`
- `发布/运维`：发布前后补 `releases.md`
- `事故/复盘`：出现线上问题或阶段性总结时补 `incidents.md`
- `沉淀/归档`：把可复用结论回写到 `articles/`、`concepts/`、`indexes/`

## 研发阶段该怎么推进

- `调研/立项`：先把问题、目标用户、成功标准、范围和非目标写清楚，不要一上来就写代码。
- `设计/评审`：先把模块、接口、数据流和关键取舍说清楚，再开始实现。
- `开发/联调`：持续更新当前状态、下一步、阻塞和测试重点，避免做着做着失去方向。
- `发布/运维`：上线前写清变更点、验证方式、回滚方案和已知风险。
- `结束/归档`：做完后回头提炼，把通用经验沉到知识库层，把项目特有内容留在项目层或归档层。

## 从零做一个系统的最小走法

1. 先在 `projects/README.md` 写清楚要解决什么问题、给谁用、做到什么算完成。
2. 如果需求开始变多，再拆 `requirements.md`，把范围、非目标、验收标准写清楚。
3. 开始实现前先补 `design.md`，把模块、接口、数据流和关键取舍说清楚。
4. 有重要取舍时就写进 `decisions.md`，不要只留在聊天记录里。
5. 开发过程中持续更新项目主页；只有当过程复杂、问题多、需要回看时，再补 `worklog.md`。
6. 准备上线时写 `releases.md`，至少说明改了什么、怎么验证、怎么回滚。
7. 做完一轮后回头提炼，把通用经验沉到知识库层，不要全留在项目层。

## 小白也能照着做的原则

- 先把问题写清楚，再写方案，再开做。
- 每个阶段至少留一个能回看的输出物，不要只靠脑子记。
- 不追求文档全，只追求当前阶段有一份主文档能指导下一步。
- 项目主页永远是第一入口；其他文件只是扩展，不是并列主入口。

## 什么时候再拆

- 项目主页已经装不下，并且某一类内容开始持续增长。
- 同一类信息会反复被查，比如需求、设计或发布说明。
- 你希望某类内容有独立历史，比如决策记录或事故复盘。

## 什么时候不拆

- 当前内容很少，项目主页一页就能看完。
- 某类内容只是一次性记录，不会复用。
- 拆出去之后反而需要来回跳转才能看明白。

## 手动流控

- 建议状态词：`idea`、`active`、`blocked`、`review`、`released`、`archived`
- 每次切状态前先读项目主页
- 如果多个文档冲突，以项目主页为准
- 如果项目结束，把可复用内容提炼回知识库，把项目特有内容保留在项目目录或归档层
