# Projects

这里放软件研发模式下的项目运行文档。

## 和知识库模式的关系

- 知识库模式是底座，负责长期沉淀和复用。
- 研发模式是叠加层，负责项目推进和阶段管理。
- 项目里的稳定结论，最终要回写到知识库层。
- 项目里的上下文、一次性决策和阶段记录，留在项目层即可。

这两层不是并排摆放，而是有演进关系：

- 外部来源先进 `raw/` 或 `inbox/`
- 当前项目正在使用和判断的内容进入 `projects/`
- 已经稳定、能复用的结论再进入 `articles/`、`concepts/`、`indexes/`
- 不再承担当前入口职责但仍有历史价值的内容进入 `archive/`

## 极简版本

极简模式最小只需要这两个东西：

- `projects/README.md`
- 根目录 `log.md`

其余文件都按需添加，不要为了完整性先建空文件。

## 这层不负责什么

- 不把项目上下文直接混进 `articles/`、`concepts/` 或 `indexes/`
- 不把可复用知识长期留在项目页里
- 不做自动流控，状态由人读项目主页后手动判断

## 推荐结构

当前设计是一个 `wiki` 只服务一个项目，所以 `projects/` 本身就是这个项目的运行层，例如：

```text
projects/
  README.md
  requirements.md
  design.md
  decisions.md
  worklog.md
  releases.md
  incidents.md
```

这些文件都按需创建，不需要一次性铺满。

## 常用文件

- `projects/README.md`：项目运行层主入口。这里放目标、状态、范围、关键链接、当前判断和下一步。
- `requirements.md`：需求说明。适合放用户问题、业务目标、范围、非目标、验收标准。
- `design.md`：方案设计。适合放架构、模块划分、接口、数据流、技术取舍。
- `decisions.md`：决策记录。适合放为什么选这个方案、不选另外一个方案、有哪些约束。
- `worklog.md`：实施记录。适合放每天推进了什么、遇到什么问题、怎么解决、还剩什么阻塞。
- `releases.md`：发布说明。适合放上线范围、变更点、验证方式、回滚方案。
- `incidents.md`：事故和复盘。适合放故障现象、影响范围、根因、修复动作和后续改进。

## 什么时候再拆

- 项目主页已经装不下，并且某一类内容开始持续增长。
- 同一类信息会反复被查，比如需求、设计或发布说明。
- 你希望某类内容有独立历史，比如决策记录或事故复盘。

## 什么时候不拆

- 当前内容很少，项目主页一页就能看完。
- 某类内容只是一次性记录，不会复用。
- 拆出去之后反而需要来回跳转才能看明白。

## 手动流控

- 建议状态词：`idea`、`active`、`blocked`、`review`、`released`、`archived`
- 每次切状态前先读项目主页
- 如果多个文档冲突，以项目主页为准
- 如果项目结束，把可复用内容提炼回知识库，把项目特有内容保留在项目目录或归档层
