# Projects

这里放活跃软件研发项目的运行文档。

## 和知识库模式的关系

- 知识库模式是底座，负责长期沉淀和复用。
- 研发模式是叠加层，负责项目推进和阶段管理。
- 项目里的稳定结论，最终要回写到知识库层。
- 项目里的上下文、一次性决策和阶段记录，留在项目层即可。

## 这层负责什么

- 需求收集
- 方案设计
- 决策记录
- 实施跟踪
- 发布说明
- 事故和复盘

## 这层不负责什么

- 不把项目上下文直接混进 `articles/`、`concepts/` 或 `indexes/`
- 不把可复用知识长期留在项目页里
- 不做自动流控，状态由人读项目主页后手动判断

## 推荐结构

每个项目建议单独一个子目录，例如：

```text
projects/
  my-service/
    README.md
    requirements.md
    design.md
    decisions.md
    worklog.md
    releases.md
    incidents.md
```

如果项目数量变多，再补一个 `projects/INDEX.md` 汇总活跃项目入口。

## 典型流程

1. 立项时先建项目目录和 `README.md`。
2. 用 `templates/project-template.md` 生成项目主页。
3. 收集需求、会议纪要、Issue、PR 和外部资料。
4. 先写清目标、范围、非目标和成功标准。
5. 再补设计、决策和实施记录。
6. 发布后写发布说明和复盘。
7. 稳定下来的结论回写到知识库层。

## 手动流控

- 建议状态词：`idea`、`active`、`blocked`、`review`、`released`、`archived`
- 每次切状态前先读项目主页
- 如果多个文档冲突，以项目主页为准
- 如果项目结束，把可复用内容提炼回知识库，把项目特有内容保留在项目目录或归档层
