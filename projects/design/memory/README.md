# 分层 Memory 设计

这页记录当前项目对“分层 memory / 系统大脑”的设计研究。

定位：

- 这是设计研究，不是当前系统已经生效的 memory 结构
- 这页用于沉淀方案、术语和边界，后续如果正式采用，再进入规则或决策层

上游：[[projects/design/README]]  
相关页：[[projects/design/memory/tools]]、[[projects/decisions]]

## 目标

这个主题要解决的核心问题只有两个：

1. 同一句话不要反复说第二次
2. 不同层的信息进入不同容器，并按规则自动上浮或下沉

更具体地说，这不是“加一个记忆功能”，而是做一套：

- 可沉淀
- 可继承
- 可升级
- 可协作
- 可冲突治理

的上下文分层系统。

## 信息分类

不是所有内容都应该进 memory。

### 1. 即时上下文

只对当前任务有用，任务结束后应该淡化。

适合：

- 当前轮临时参数
- 当前报错日志
- 本轮讨论里的中间假设

### 2. 稳定偏好

会长期影响协作方式。

适合：

- 输出风格偏好
- 决策方式偏好
- 协作习惯
- 反复强调的工作原则

### 3. 项目事实

围绕某个项目、产品、模块持续有效的共识。

适合：

- 项目目标
- 模块边界
- 已确认约束
- 术语定义
- 历史原因

### 4. 决策与制度

这不是普通信息，而是裁决结果。

适合：

- 冲突升级规则
- 自动沉淀边界
- 人工确认边界
- 优先级顺序

## 推荐分层

建议采用 5 层，而不是只有一层 memory。

### L0：Session Working Memory

作用：承接当前会话内的临时上下文  
生命周期：短

适合：

- 本轮目标
- 当前待办
- 尚未确认的推断
- 本次会话的中间结论

### L1：User Memory

作用：记录用户稳定偏好  
生命周期：长

适合：

- 输出风格偏好
- 决策方式偏好
- 协作习惯
- 反复强调的工作原则

### L2：Project / Workspace Memory

作用：沉淀项目共享脑  
生命周期：中长

适合：

- 项目目标
- 已确认约束
- 架构边界
- 术语定义
- 多任务重复引用的共识

### L3：Decision / Policy Memory

作用：沉淀高优先级规则  
生命周期：长，且需要版本化

适合：

- 决策结论
- 升级机制
- 例外处理原则
- 冲突裁决规则

### L4：Knowledge Artifact

作用：把高价值 memory 固化成可引用资产

形式可以是：

- ADR
- 决策记录
- 项目约定文档
- 术语表
- 工作流规则文档

## 层间流动

建议默认链路：

`对话内容 -> 候选事实 -> 分类 -> 置信度判断 -> 写入对应层 -> 周期性压缩或升级`

基本规则：

1. 默认先落 L0
2. 满足条件后再晋升到 L1 或 L2
3. 被裁决的内容再进入 L3
4. 高价值的 L2/L3 再固化到 L4

## 写入路由

### 进 Agent Prompt / System Context

只放高频、必须即时生效、体积小、优先级高的内容。

适合：

- 当前任务目标
- 当前项目 ID
- 当前适用 policy 摘要
- 用户关键偏好摘要
- 当前会话必须继承的最近结论

### 进 Memory Store

放可复用、非瞬时、未来有价值的内容。

分两类：

- 用户级 memory
- 项目级 memory

### 进 Decision / Policy

放会改变后续执行规则的内容。

适合：

- 冲突如何升级
- 哪些信息允许自动沉淀
- 哪些必须人工确认
- 优先级顺序

## 自动沉淀规则

建议做一个“沉淀判定器”，在每轮结束或任务结束时跑一次。

判断维度：

- 是否稳定
- 是否可复用
- 是否影响未来输出
- 是否已确认
- 是否属于用户、项目或制度

建议映射：

- 临时任务信息 -> L0
- 用户长期偏好 -> L1
- 项目背景与共识 -> L2
- 拍板后的规则 -> L3
- 需要可审计归档 -> L4

## 冲突处理

memory 不应该是覆盖式，而应带状态。

建议每条记忆至少包含：

- `scope`
- `status`
- `confidence`
- `source`
- `last_verified_at`
- `owner`
- `supersedes`

冲突处理流程：

1. 标记 `conflicted`
2. 生成冲突摘要
3. 等待决策
4. 决策后把旧记录标记为 `deprecated`，新记录改为 `confirmed`

## 优先级顺序

建议固定读取顺序：

`Policy > Project Memory > User Memory > Session Memory > Raw Chat History`

## 可落地模块

建议拆成这些模块：

1. `Capture`
2. `Classifier`
3. `Consolidator`
4. `Conflict Resolver`
5. `Retriever`
6. `Publisher`

## 最小可执行版本

V1 可以先只做 4 层：

- `session_memory`
- `user_memory`
- `workspace_memory`
- `decision_memory`

V1 规则：

1. 默认都先入 `session_memory`
2. 重复出现或明确声明的，晋升到 `user` 或 `workspace`
3. 影响多人协作或流程的，进入 `decision`

V1 注入顺序：

- `decision > workspace > user > session`

## 当前结论

这套设计真正要解决的不是“记忆存储”，而是：

- 分类能力
- 优先级能力
- 晋升能力
- 冲突治理能力
- 可审计能力

一句话定义：

把多轮对话中的临时信息、稳定偏好、项目共识和决策规则分层沉淀，按优先级自动注入后续任务，使系统从“被动记录日志”升级为“主动继承背景、统一协作口径、避免重复沟通”的工作大脑。
