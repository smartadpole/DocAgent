# 模板反哺规则

这页回答：当某个项目文档库在真实使用中进化出更好的结构、流程或模板时，如何把它反哺回这套 wiki 模板。

主入口：[[governance/README]]

## 这页负责什么

- 定义项目文档库反哺模板库的判断顺序
- 区分“项目事实”和“系统能力”
- 给跨项目复用的结构、流程、规则和模板找到正确落点
- 避免把下游项目的业务信息、技术结论或临时状态复制回模板

## 这页不负责什么

- 不记录某个项目自己的业务决策，那些留在该项目的 [[projects/decisions]]
- 不记录项目级稳定事实，那些留在该项目的 [[projects/memory/README]]
- 不替代 [[WORKFLOW]] 的日常执行流程
- 不替代 [[AGENTS]] 的硬约束

## 反哺判断

从下游项目抽取变化时，先按这组问题判断：

1. 这项变化是否已经在真实项目推进中证明有用。
2. 它是否去掉项目名、业务名、具体技术选型后仍然成立。
3. 它是否会影响多个未来项目的结构、读取顺序或写法。
4. 它是否只是某个项目的一次性事实、状态、会议结论或实现取舍。

只有前 3 条成立，且第 4 条不成立，才进入模板反哺。

## 反哺路由

- 新的硬约束，写进 [[AGENTS]]
- 新的推进顺序，写进 [[WORKFLOW]]
- 新的规则、优先级或自动沉淀边界，写进 [[POLICY]]
- 新的共享背景或系统定位，写进 [[BRAIN]]
- 新的项目层结构，写进 [[projects/STRUCTURE]]，并按需补入口页
- 新的可复制写法，写进 `templates/`
- 新的通用方法，写进 `concepts/` 或 `articles/`
- 本轮反哺动作，写进 [[log]]

## 最小执行顺序

1. 先读模板库的 [[README]]、[[INDEX]]、[[BRAIN]]、[[POLICY]] 和 [[WORKFLOW]]。
2. 再读下游项目里发生变化的入口页、结构页、规则页和日志。
3. 列出候选反哺项，并给每一项标注“保留项目内 / 反哺模板 / 暂不处理”。
4. 对反哺项做脱敏和抽象：删除项目名、业务名、具体数据、具体结论和一次性状态。
5. 按反哺路由更新模板库，不复制下游项目正文。
6. 回看入口页、结构页、链接和模板，确认没有缺失目标或第二份模板正文。
7. 在 [[log]] 记录这次反哺的主题、来源项目、抽象后的系统变化和影响页面。
8. 提交模板库改动；下游项目的预存脏改动不并入模板反哺提交。

## 当前反哺口径

- 下游项目可以各自演化，但模板库只吸收“系统能力”，不吸收“项目内容”。
- 如果某个下游项目的结构已经明显优于模板，优先把它抽成通用骨架和规则，再回写模板。
- 如果一项变化仍然依赖项目阶段、业务范围、团队取舍或技术栈选择，就留在下游项目，不进入模板。
- 如果反哺项本身改变模板维护方式，必须同步更新 [[AGENTS]]、[[WORKFLOW]]、[[POLICY]] 或 [[BRAIN]]。
