# 模板反哺规则

这页回答：当某个项目文档库在真实使用中进化出更好的系统层信息时，如何把它反哺回这套 wiki 模板。

主入口：[[governance/README]]

## 这页负责什么

- 定义项目文档库反哺模板库的判断顺序
- 区分“项目材料”和“系统层信息”
- 给跨项目复用的结构、流程、规则、记忆路由、写法和模板找到正确落点
- 避免把下游项目的业务信息、技术结论或临时状态复制回模板

## 这页不负责什么

- 不记录某个项目自己的业务决策，那些留在该项目的 [[projects/decisions]]
- 不记录项目级稳定事实，那些留在该项目的 [[projects/memory/README]]
- 不替代 [[WORKFLOW]] 的日常执行流程
- 不替代 [[AGENTS]] 的硬约束

## 系统层信息范围

反哺时不要只看“结构性材料”。下面这些都属于系统层信息，只要在下游项目中形成了规则、流程或稳定写法，就默认进入反哺候选：

1. **结构性信息**：目录、入口页、主页面、子页类型、上下游关系、单一信息源、项目层文件职责。
2. **流程性信息**：读取顺序、推进顺序、收尾流程、批处理流程、会议流程、从设计到开发的拆解链路。
3. **规则性信息**：硬约束、禁止项、冲突升级、自动写入边界、同步要求、提交要求、格式要求。
4. **记忆系统信息**：[[BRAIN]]、[[POLICY]]、[[projects/memory/README]]、[[projects/trace]]、[[log]] 和 `workspace-memory` 的路由边界。
5. **写法与格式信息**：标题格式、字段顺序、编号方式、强调方式、记录骨架、同日记录复核、内部链接格式。
6. **模板信息**：`templates/` 下可复制骨架、模板入口、模板使用说明，以及“模板正文唯一”的约束。
7. **自动化契约信息**：frontmatter 字段、状态轴、阶段轴、agent 最小读取集、交付前检查、机器可读镜像页。
8. **治理说明信息**：某类系统信息应该由哪一页裁定、哪一页执行、哪一页记录历史。

一句话规则：

- 只要它是规则，就默认需要反哺。
- 如果规则里夹带项目事实，反哺时抽掉事实，只保留可复用规则。
- 如果规则只因某个项目的一次性业务选择才成立，保留在该项目；但也要记录“为什么不反哺”。

## 项目材料范围

下面这些默认不是模板反哺内容：

- 某个项目的业务范围、用户角色、产品决策、技术栈拍板和阶段状态
- 某次会议的具体结论、行动项、参会人和项目待办
- 某个现成工程的具体页面、表名、字段、bug 和复用结论
- 某份原始材料的摘要、出处、数据、样例和项目证据
- 只对当前项目长期有效的事实

如果这些内容背后出现了可复用规则，例如“会议页只引用设计专题，不重复正文”，则规则要反哺，项目事实不反哺。

## 反哺判断

从下游项目抽取变化时，先按这组问题判断：

1. 它是不是规则、流程、结构、记忆路由、写法、模板或自动化契约。
2. 它是否已经在真实项目推进中证明有用。
3. 它是否去掉项目名、业务名、具体技术选型后仍然成立。
4. 它是否会影响多个未来项目的结构、读取顺序、写法或 agent 行为。
5. 它是否只是某个项目的一次性事实、状态、会议结论或实现取舍。

只要第 1 条成立，默认进入模板反哺候选；第 5 条成立时，只抽象其中的规则，不复制项目事实。

## 反哺路由

- 新的硬约束，写进 [[AGENTS]]
- 新的推进顺序，写进 [[WORKFLOW]]
- 新的规则、优先级或自动沉淀边界，写进 [[POLICY]]
- 新的共享背景或系统定位，写进 [[BRAIN]]
- 新的项目层结构，写进 [[projects/STRUCTURE]]，并按需补入口页
- 新的记忆路由，优先写 [[POLICY]]，再同步 [[WORKFLOW]] 和 [[AGENTS]]
- 新的日志、trace 或会议写法规则，写进对应 `governance/*-rules`、[[WORKFLOW]]、[[AGENTS]] 和 `templates/`
- 新的可复制写法，写进 `templates/`
- 新的通用方法，写进 `concepts/` 或 `articles/`
- 本轮反哺动作，写进 [[log]]

## 最小执行顺序

1. 先读模板库的 [[README]]、[[INDEX]]、[[BRAIN]]、[[POLICY]] 和 [[WORKFLOW]]。
2. 再读下游项目里发生变化的入口页、结构页、治理页、规则页、模板页、记忆页和日志。
3. 列出候选反哺项，并给每一项标注“系统规则 / 系统结构 / 系统流程 / 系统模板 / 系统记忆路由 / 项目材料”。
4. 对反哺项做脱敏和抽象：删除项目名、业务名、具体数据、具体结论和一次性状态。
5. 系统规则默认反哺；如果决定不反哺，必须能说清它为什么只是项目材料。
6. 按反哺路由更新模板库，不复制下游项目正文。
7. 回看入口页、结构页、链接、规则页和模板，确认没有缺失目标或第二份模板正文。
8. 在 [[log]] 记录这次反哺的主题、来源项目、抽象后的系统变化和影响页面。
9. 提交模板库改动；下游项目的预存脏改动不并入模板反哺提交。

## 当前反哺口径

- 下游项目可以各自演化，但模板库吸收全部系统层信息，不吸收项目材料。
- 系统层信息不只包括结构，也包括流程、规则、记忆系统、格式治理、模板和自动化契约。
- 如果某个下游项目的规则已经明显优于模板，优先把它抽成通用规则，再回写模板。
- 如果一项变化仍然依赖项目阶段、业务范围、团队取舍或技术栈选择，就只保留它背后的通用规则，不带入具体项目结论。
- 如果反哺项本身改变模板维护方式，必须同步更新 [[AGENTS]]、[[WORKFLOW]]、[[POLICY]]、[[BRAIN]] 和本页。
